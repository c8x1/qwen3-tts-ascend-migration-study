优化器、Checkpoint 与导出

把 Accelerate ownership、AdamW 更新、repo ID 与本地目录冲突、主进程 safetensors 导出及 resume 缺口分开陈述。

学习目标

  • 区分 Accelerate 的已核验调用与 process-local/wrapper 静态推断
  • 解释默认 Hugging Face repo ID 与本地 copytree/config open 的静态冲突
  • 按主进程导出顺序定位 config rewrite、speaker row 3000 与 safetensors 边界

前置知识

  • 熟悉 PyTorch 单卡张量与训练循环
已核验项目声明静态推断待真机验证

Accelerate ownership：verified 与 pending 分开

状态：已核验 已核验 ownership call：qwen3tts.model、optimizer、train_dataloader 一起进入 accelerator.prepare，并使用返回的 model、optimizer、train_dataloader。

状态：静态推断 静态推断 / pending execution：module global target_speaker_embedding 是每个 Python interpreter 的状态；实际 process-local 值、不同进程首先看到的 embedding、DataLoader runtime sharding、prepared wrapper 的 direct attributes/custom method 行为都依赖启动方式与 Accelerate 版本。源码不证明跨进程共享，也不证明必然失败。

Checkpoint / resume 缺口

状态：静态推断 这个 epoch 产物是供 CustomVoice 装载的模型目录与 model.safetensors；safetensors 不等于 optimizer/RNG resume。固定脚本没有保存 optimizer state、scheduler state、RNG、sampler position 或 Accelerator training state，也没有 resume/load_state 路径。该结论是 fixed-tree audit inference，不证明仓库外其他系统不存在。
