# Qwen3-TTS 到 Ascend 910B 训练迁移参考研究与知识站设计

日期：2026-07-16

## 1. 背景

用户准备学习如何将 Qwen3-TTS 训练工程从 NVIDIA GPU 迁移到华为 Ascend 910B 系列。用户熟悉 PyTorch 单卡训练，但尚不熟悉分布式训练与 Ascend 生态。当前可用环境基线是 CANN 8.5.2 与 PyTorch 2.7.1，但允许按项目兼容性新建环境或调整版本。

学习范围覆盖完整训练链路：预训练、SFT、推理验证、单机 8 卡与多机多卡。规模化训练允许并优先采用 MindSpeed/MindSpeed-LLM 等华为训练生态组件，不把纯 torch-npu 最小迁移作为主路线。

本项目的任务是研究和教学，不是实际改造 Qwen3-TTS：系统检索公开成熟项目，选择最有参考价值的工程，固定版本并获取源码，完整走读其已有 Ascend 适配与训练设计，最终制作一组详尽的中文 HTML 学习材料。

## 2. 目标与非目标

### 2.1 目标

1. 系统检索与 Qwen3-TTS、Ascend 910B、MindSpeed 和语音生成训练有关的公开项目及官方资料。
2. 通过统一评分和证据标准选择一个主参考项目，并用少量专项参考项目补足其缺口。
3. 固定并获取所选项目源码，记录来源、commit、许可证、环境和适用边界。
4. 建立主项目的目录、符号、配置、调用链、训练阶段和硬件相关点覆盖矩阵。
5. 建立 Qwen3-TTS 模块到参考项目实现、Ascend 迁移知识和未来验证项的映射。
6. 生成可离线浏览、可检索、可追溯的多页中文 HTML 知识站，使具备 PyTorch 单卡基础的读者能够为未来真实迁移做好知识准备。

### 2.2 非目标

1. 不在本阶段实际修改 Qwen3-TTS 源码。
2. 不宣称在 Ascend 910B 真机上跑通训练或达到性能目标。
3. 默认不下载模型权重、训练数据集或其他大体积非必要产物。
4. 不为了给出唯一答案而把不完整或仅有宣传声明的项目描述为成熟方案。
5. 不复制许可证不允许再分发的源码或受限内容。

## 3. 总体方案

采用“双轴组合”方案，而不是依赖单一项目：

- 目标基准：Qwen3-TTS 官方源码、论文或技术资料，用于还原目标架构与 NVIDIA 训练依赖。
- 主参考项目：在“Qwen3-TTS 架构接近度”和“Ascend 训练成熟度”上综合得分最高的项目，进行完整源码走读。
- 训练专项参考：MindSpeed/MindSpeed-LLM、Qwen 系列 Ascend 训练项目，用于补足并行、性能、多机和容错。
- 语音专项参考：Ascend 上的 TTS、语音生成、音频 Tokenizer 或 Codec 项目，用于补足音频数据链路和质量评测。
- 官方依据：CANN、torch-npu、HCCL、MindSpeed 等官方文档，用于核验版本、接口和迁移方法。

参考项目不会被混合成一套新代码。知识站使用映射表明确每个项目负责解释什么、证据来自哪里、哪些知识可迁移以及哪些结论仍需真机验证。

## 4. 调研范围与候选发现

检索范围包括：

- 华为官方文档、Ascend 社区、ModelZoo、MindSpeed 与相关官方仓库；
- Qwen/Alibaba 官方仓库、论文、技术报告和模型页面；
- GitHub、Gitee、ModelScope、Hugging Face 等公开代码与模型平台；
- 组织级开源项目、可核验的 PR、Issue、发布记录、配置和训练日志；
- 论文与高质量技术文章，用于补充背景或发现线索。

“全网”在本项目中指多渠道系统检索、记录检索式并交叉验证，不意味着可以穷尽无法索引、已删除、私有或受访问控制的内容。

候选发现阶段保存检索式、来源 URL、访问日期、维护状态、初筛理由和排除理由，以便复查调研覆盖面。

## 5. 候选评分与证据标准

### 5.1 评分模型

候选项目按 100 分评估：

| 维度 | 权重 | 核心问题 |
| --- | ---: | --- |
| Ascend 改造完整度 | 30 | 是否包含训练而非仅推理；是否覆盖设备、算子、精度、分布式、Checkpoint 和验证 |
| 与 Qwen3-TTS 的架构接近度 | 25 | Transformer 主干、生成范式、音频 Tokenizer/Codec、多阶段训练与数据流是否相近 |
| 大规模训练成熟度 | 20 | 是否支持单机 8 卡、多机、MindSpeed、并行策略、性能和容错 |
| 可复现证据 | 15 | 是否提供固定版本、环境矩阵、启动脚本、日志、产物、已知问题和维护记录 |
| 文档、许可与可维护性 | 10 | 文档是否完整、许可证是否清晰、代码是否可维护 |

总分用于形成候选排序，但不会替代硬性事实核验。若两个主候选优劣取决于价值判断，必须向用户展示证据和取舍后再决定。

### 5.2 来源等级

- S 级：华为、Qwen/Alibaba 等官方源码、官方文档、版本说明和提交记录。
- A 级：组织级开源仓库、官方 ModelZoo、可核验的 PR/Issue、训练配置和日志。
- B 级：具有完整源码和复现材料的可信社区项目。
- C 级：博客、论坛和二手转载；只能用于发现线索，不能独立支撑关键结论。

### 5.3 结论状态

每个重要结论必须标记为以下一种：

- 已证实：源码或官方材料直接支持。
- 项目声明：项目作者声称支持，但没有足够的独立运行证据。
- 分析推断：根据架构与源码得出的迁移判断。
- 待真机验证：必须在 Ascend 910B 集群上确认。

## 6. 研究与内容生产流程

### 6.1 全网候选发现

并行检索官方生态、代码平台、模型平台与技术材料，形成去重候选清单和检索记录。

### 6.2 候选审计与选型

对候选项目进行源码级核验，确认它是训练工程还是推理演示、Ascend 支持是实际代码还是 README 声明，并输出评分、排名、证据和淘汰原因。

### 6.3 固定并获取参考源码

拉取主项目及必要专项项目，固定到可追溯 commit；记录远端地址、分支、子模块、许可证、依赖、推荐环境和获取日期。知识站中的远程源码链接优先指向固定 commit，避免上游更新造成定位漂移。

### 6.4 建立源码知识图谱

盘点目录、训练入口、配置继承、数据管线、模型构造、Forward/Loss、优化器、混合精度、并行初始化、Checkpoint、推理和评测，形成以下关联：

`文件 → 类/函数/配置 → 调用方 → 训练阶段 → Ascend 相关点 → 讲解页面`

### 6.5 建立迁移映射

以 Qwen3-TTS 为目标侧，以主项目和专项项目为证据侧，逐项覆盖：

- 环境和依赖；
- CUDA/NCCL 到 CANN/HCCL；
- 设备 API、随机数、AMP、BF16/FP16 和损失缩放；
- 不支持算子、融合算子、自定义扩展和编译链；
- 数据和音频 I/O、动态 Shape、Bucketing 与 Host 性能；
- 单机 8 卡和多机并行；
- Checkpoint、权重转换、推理、质量评测；
- 正确性、性能和稳定性验证。

### 6.6 生成与审计静态知识站

将研究数据和走读内容渲染为多页静态 HTML，生成导航、全文搜索、上下页、源码链接、交叉引用和索引，最后执行完整性、链接和渲染检查。

## 7. 知识站信息架构

页面数量不预先封顶，由源码模块与覆盖清单决定。初步规模预计不少于 25–40 个主题页；主项目复杂时继续拆分，不因追求固定篇数而合并细节。

### 7.1 导航、选型与基础

- 首页、推荐阅读路径、术语表和全站索引；
- 候选项目排行榜、证据卡、评分依据与淘汰原因；
- Qwen3-TTS 目标架构、训练阶段和数据流；
- Ascend 910B、CANN、torch-npu、HCCL 与 MindSpeed；
- DDP、TP、PP、CP、ZeRO、显存估算和多机训练基础。

### 7.2 主项目深度走读专题

每个重要子系统独立成页；复杂子系统继续按文件或调用链拆分：

- 仓库结构、版本、安装和构建；
- 训练入口与配置系统；
- 数据集、音频预处理、Tokenizer/Codec；
- 模型组件、Forward、Loss 和训练循环；
- 混合精度、算子替换和编译；
- 并行策略、多机启动、通信与容错；
- Checkpoint、推理、评测、性能调优与故障处理。

### 7.3 迁移映射专题

迁移映射不放在单一页面，而是按未来 Qwen3-TTS 迁移工作包拆页，覆盖第 6.5 节列出的每个方向。每页说明目标模块、参考实现、可复用方法、版本约束、证据等级、风险和未来验证步骤。

### 7.4 可检索附录

- 源码文件索引；
- 类、函数和配置项索引；
- NVIDIA 专属点与 Ascend 对照表；
- 官方来源、固定版本链接、访问日期和证据等级；
- 未知项、冲突证据、真机待验证项和常见错误手册。

## 8. 单页内容模板

页面的视觉系统、响应式三栏布局与侧栏交互由 [`2026-07-16-knowledge-site-template-design.md`](2026-07-16-knowledge-site-template-design.md) 进一步约束。

每个主题页根据适用性包含：

1. 学习目标与前置知识；
2. 该模块在完整调用链中的位置；
3. 关键源码文件、类、函数和配置；
4. 逐段或逐调用链解释；
5. 数据、张量形状、状态或控制流；
6. NVIDIA 与 Ascend 对照；
7. 版本、依赖与适用边界；
8. 常见故障和诊断路径；
9. 验证方法与预期观察；
10. 来源、证据状态与延伸阅读。

页面使用中文解释并保留必要的英文术语。源码不会大段复制，而是使用短片段、符号定位和固定 commit 链接，以保持可追溯性并遵守许可证与引用规范。

## 9. 防遗漏机制

写作前先生成仓库清单和源码覆盖矩阵。主项目中的关键文件、训练阶段和硬件相关点必须满足以下任一条件：

1. 映射到至少一个讲解页面；
2. 标记为重复、生成文件、第三方代码或与训练无关，并记录排除理由；
3. 标记为未知或待进一步核验，不允许静默遗漏。

Qwen3-TTS 目标侧采用相同机制，确保关键训练模块进入迁移映射。最终执行源码覆盖审计、事实状态审计、来源检查、站内死链检查和页面结构检查。

## 10. 异常与不确定性处理

- 没有完全匹配的成熟项目：明确报告缺口，以主项目加专项项目补足，不降低证据标准。
- 官方文档与源码冲突：优先固定版本源码和对应版本文档，并单列冲突。
- 只有 Ascend 支持声明：标记为“项目声明”，不写成已跑通。
- 项目版本较旧：同时呈现项目原始环境、当前 CANN 8.5.2/PyTorch 2.7.1 基线和推荐环境，不擅自假定兼容。
- 仓库失效或受限：优先官方镜像；不能合法获取时只保留元数据和链接。
- 缺少 910B 真机：不宣称训练或性能验证完成。
- 关键来源相互矛盾、主项目候选难分高下、许可证有风险、版本选择会实质改变结论，或结论依赖未经验证的关键假设：暂停并优先询问用户。
- 普通检索、排版、索引和不改变结论的细节：自主推进，避免无谓中断。

## 11. 工作区与交付物

项目使用一个新建的公开 GitHub 仓库持续保存研究过程与交付物。首个提交包含设计规格和仓库基础规则；后续在候选调研、项目选型、源码走读、迁移映射和知识站生成等可审阅节点提交并推送。公开仓库不提交模型权重、数据集、密钥、访问令牌、可视化临时文件或许可证不允许再分发的内容。

计划目录：

```text
research/                 检索记录、候选评分、来源、证据卡和覆盖矩阵
references/               固定版本的主项目与专项项目源码
site/                     可直接打开的多页中文 HTML 知识站
site/assets/              本地样式、搜索索引、图和其他静态资源
docs/superpowers/specs/   设计说明
docs/superpowers/plans/   实施计划
```

核心交付物：

- 候选项目清单、排行榜、证据与淘汰原因；
- 主项目和专项项目固定版本源码；
- Qwen3-TTS 目标架构走读；
- Ascend 与分布式训练前置课程；
- 主项目多页面深度走读；
- 多页面迁移映射；
- 源码、符号、配置与硬件相关点索引；
- 环境兼容矩阵、故障手册和未来真机实验清单；
- 可追溯来源与证据状态。

HTML 的阅读、导航和搜索使用本地静态资源，不依赖在线服务。外部来源链接保留用于事实核验。

## 12. 验收标准

1. 检索记录、候选评分、来源和淘汰理由齐全。
2. 主项目及专项项目固定到 commit，并记录许可证和环境。
3. 主项目关键源码文件全部进入覆盖矩阵，均有页面或排除理由。
4. Qwen3-TTS 关键训练模块全部进入迁移映射矩阵。
5. 每项重要迁移结论都有来源和证据状态。
6. 知识站提供目录、全文搜索、交叉引用、上下页和离线阅读能力。
7. 站内无死链；关键外链在生成时可访问；远程代码定位使用固定版本。
8. 页面通过自动结构检查，并抽样人工核对源码解释与引用。
9. 提供从 PyTorch 单卡基础出发的推荐学习路径。
10. 提供单卡冒烟、单机 8 卡和多机多卡三个未来实验清单。
11. 明确区分文档完成、源码静态核验和 Ascend 910B 真机验证，禁止混淆三者。

## 13. 已确认决策

- 目标硬件：华为 Ascend 910B 系列。
- 训练范围：预训练、SFT、推理验证；覆盖单机 8 卡和多机多卡知识。
- 环境：以 CANN 8.5.2、PyTorch 2.7.1 为当前基线，但允许为兼容性推荐其他环境。
- 学习起点：熟悉 PyTorch 单卡，不假设掌握分布式和 Ascend。
- 技术路线：允许并优先采用 MindSpeed/MindSpeed-LLM 等华为训练生态。
- 交付边界：研究成熟参考工程并制作走读资料，不实施 Qwen3-TTS 源码迁移。
- 项目策略：一个主参考项目加少量专项参考项目。
- 内容粒度：多页面深度走读与迁移映射，页数由覆盖完整性决定。
- 决策原则：拿不准且影响结论时优先询问用户。
- 发布方式：新建公开 GitHub 仓库，并在研究过程中持续提交与推送可审阅产物。
