# Research Methodology

本文件定义 Phase 1 候选发现、证据记录、评分和升级决策的统一方法。三份 CSV 是后续研究任务的事实源；叙述性报告必须能够回指这些记录。

## Channel inventory

每轮候选发现至少覆盖与研究问题相关的以下必需渠道，并在 `search-log.csv` 分别记录：

1. 华为、Qwen/Alibaba 等官方源码、产品文档、版本说明和提交记录；
2. GitHub；
3. Gitee；
4. ModelScope；
5. Hugging Face；
6. 可核验的 PR、Issue、训练配置、日志、论文和高质量技术材料。

“全网”表示多渠道系统检索、记录检索式并交叉验证，不表示能够穷尽未索引、已删除、私有或受访问控制的内容。

## Exact query logging

每次实际查询占 `search-log.csv` 一行。`query` 原样保存提交给渠道的完整检索式，不事后改写；`channel` 记录实际入口，`run_at` 使用带时区的 ISO 8601 时间，`result_count` 记录当时可观察的结果数，`accepted_ids` 用 `|` 连接本次接纳的候选 ID，`notes` 记录筛选范围、分页边界、访问限制或重复项。查询式或过滤条件发生变化时必须新建记录。

## Inclusion rules

候选进入台账须满足以下条件：

- 公开可核验，且能够记录稳定的仓库或项目 URL；
- 与 Qwen3-TTS 架构、Ascend 训练、MindSpeed/分布式训练或语音生成训练至少一个研究轴直接相关；
- 提供可审查的源码、配置、提交、PR/Issue、训练日志或官方材料，而不只是无法核验的宣传摘要；
- 其训练阶段、硬件支持范围、维护状态和许可证能够被记录，未知项可明确标记并继续核验。

训练专项或语音专项项目可作为补缺候选，不要求单个项目覆盖全部研究轴。

## Exclusion rules

下列项目排除，并在 `candidates.csv` 的 `exclusion_reason` 记录可复查理由：

- 与目标训练链路无实质关系，或仅提供通用推理演示且没有可迁移的专项价值；
- 只有二手宣传或 README 声明，无法找到对应源码、配置或官方证据；
- 内容已失效、私有或受限，且没有可合法核验的官方镜像或元数据；
- 是已有候选的无实质差异镜像、分叉或重复条目；
- 许可证明确禁止本研究所需的检查或再分发方式，且无法通过只保留元数据和链接合规处理。

排除不等于删除：记录保留并将状态设为 `rejected`，以避免后续重复发现。

## Source levels

- **S 级**：华为、Qwen/Alibaba 等官方源码、官方文档、版本说明和提交记录。
- **A 级**：组织级开源仓库、官方 ModelZoo、可核验的 PR/Issue、训练配置和日志。
- **B 级**：具有完整源码和复现材料的可信社区项目。
- **C 级**：博客、论坛和二手转载；只能发现线索，不能单独支撑关键结论。

## Claim states

每项重要结论必须明确标记为以下一种状态：

- **已证实**：源码或官方材料直接支持。
- **项目声明**：项目作者声称支持，但没有足够的独立运行证据。
- **分析推断**：根据架构与源码得出的迁移判断。
- **待真机验证**：必须在 Ascend 910B 集群上确认。

证据增强或出现冲突时更新状态，不得把“项目声明”或“分析推断”静默改写为“已证实”。

## Score rubric

候选按 100 分制评分，各维度只在其范围内取整数；`total` 必须等于五项之和：

| Dimension | Maximum | Evaluation focus |
| --- | ---: | --- |
| Ascend 改造完整度 (`ascend_completeness`) | 30 | 训练而非仅推理；设备、算子、精度、分布式、Checkpoint 和验证覆盖 |
| Qwen3-TTS 架构接近度 (`architecture_proximity`) | 25 | Transformer 主干、生成范式、音频 Tokenizer/Codec、多阶段训练和数据流 |
| 大规模训练成熟度 (`scale_maturity`) | 20 | 单机 8 卡、多机、MindSpeed、并行策略、性能和容错 |
| 可复现证据 (`reproducibility`) | 15 | 固定版本、环境矩阵、启动脚本、日志、产物、已知问题和维护记录 |
| 文档、许可与可维护性 (`docs_license`) | 10 | 文档完整度、许可证清晰度和代码可维护性 |

总分用于排序，不能替代硬性事实核验。每个分数必须能回指 `evidence_ids` 中的来源。

## Duplicate handling

同一上游仓库的别名、镜像、旧组织地址和无实质差异 fork 合并为一个候选，以权威上游 URL 为主记录。镜像可作为独立来源写入 `source-ledger.csv`，但不重复计分。若 fork 含有独立且相关的 Ascend 训练改造，则保留为单独候选，并在 notes 中说明与上游的关系。项目改名或迁移不得复用另一个实体的 ID。

## Fixed-revision policy

发现阶段可暂用移动分支定位；进入 `audited` 或 `recommended` 前必须将 `revision` 固定为可追溯的完整 commit。标签要解析到具体 commit，子模块也记录其固定 revision。报告和远程源码链接优先指向固定 commit；上游后续更新需要作为新审计记录评估，不能静默移动既有证据基线。

## License policy

记录仓库在固定 revision 上声明的许可证标识或原文；无法确认时写明 `UNKNOWN`，不得猜测。默认只提交研究所需的元数据、短引用和固定版本链接，不提交许可证禁止再分发的源码、权重、数据集或其他受限内容。许可证风险会改变候选选择、技术路线或交付方式时，按重大决策升级。

## Search saturation

只有在所有必需渠道均已查询，且连续两轮 query-family pass 都没有新增符合 inclusion rules 的候选时，才可声明候选发现达到搜索饱和。query family 是围绕同一研究轴的一组同义词、平台限定和中英文变体；每轮的具体查询仍须逐条写入 `search-log.csv`。新增候选会将连续无新增计数清零。

## Major-decision escalation

出现以下任一情形必须暂停并请求用户确认：主参考项目或专项组合将改变、技术路线将改变、许可证处理存在实质风险、关键来源相互矛盾、主候选难分高下、版本选择会实质改变结论，或核心结论依赖未经验证的关键假设。普通检索、格式、索引和不改变结论的细节采用保守方案继续，并记录到 `IMPLEMENTATION_NOTES.md`。

## CSV serialization

三份 CSV 固定使用 UTF-8（无 BOM）、逗号分隔、LF 行尾和 brief 定义的精确列顺序；需要转义时使用标准 CSV 双引号。多值 ID 字段使用 `|`，未知单值留空，禁止用逗号承载列表以免破坏列结构。这是当前跨平台、标准库可直接读取的保守格式选择。
